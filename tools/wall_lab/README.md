# Dungeon wall lab

The dungeon wall autotiler is developed here, not in `source/`. It is a rule about
how a tile *looks* next to its neighbours, and it only means anything next to the
actual Sprout Lands Dungeon Pack sheet — so it is authored as a browser page you
can stare at, and only ported to C++ once it is right.

```
scenarios_template.html   the lab. THE RULE LIVES HERE (in its <script>).
splice.py                 template + base64 sheets -> scenarios.html (publishable)
corrections_round3.js     tiles the user marked wrong/right, transcribed by hand
gen_cases.js              corrections -> ../../tests/js/wall_cases.json
*_b64.txt                 tilesheets, inlined because the artifact CSP blocks all
                          external requests
```

## The page

Three tabs, one engine:

- **Scenarios** — a deterministic gallery of wall/floor configurations. Click any
  cell to see its three layers (floor / wall / overlay) and correct any of them;
  or hit "Looks right" to approve the whole render. Exports `{corrections, approved}`.
- **Dungeon** — a generated room-and-corridor map rendered with the same rule, in
  both boulder and brick. This is where the rule meets geometry the 5×5 cards can't
  produce.
- **Hedge** — the overworld hedge autotiler, unrelated to dungeon walls.

Publish with `python tools/wall_lab/splice.py`, then upload the generated
`scenarios.html`. The live copy is:

    https://claude.ai/code/artifact/4d9c8a7f-11e9-491d-bdb2-04eecad1d746

Publishing a *path* that has never been published before mints a **new** URL, and
this file moved here from a scratch directory — so republishing must target the
existing artifact by URL explicitly, or the link above goes stale and the user's
saved corrections (kept in that page's localStorage) are stranded on the old one.

## The rule, briefly

Mask bits `N=1 E=2 S=4 W=8`; a **set bit means FLOOR** on that side. Walls are 2.5D
walk-behind: they read two tiles tall and the player passes behind the top.

- A **south-facing face** (`em & 4`) keeps its tile untouched.
- Otherwise, if the structure **continues below** (`wbDropsN`: `isCap ||`
  floor at either diagonal-below), the cell drops its N bit. That test — *does this
  wall keep going down* — is the whole rule. It is **not** a real-vs-bleed test;
  that hypothesis was wrong.
- **Top rims peel upward**: a wall with floor to the north casts its rim tile onto
  the floor cell above it as an **overlay**, which the player walks behind.

> The mechanics, in rule order: a south face (`em & 4`) is untouched; a side
> junction (`fcl`/`fcr`, both sides → `faceBoth`) wins; otherwise the N contact
> peels up as an overlay and the wall renders with `em2 = em & ~1`. Corner caps
> keep their identity (variant iff `em2 ≠ 0`); a cap with floor both sides leans
> `capE2`/`capW2` toward the wall continuing diagonally below; plain caps drop
> r1→r0, r3→r2w, r9→r8. A side contact whose run ends going down rounds into a
> corner (`!isOpen(x±1, y+1)` — the row *below* only), with a leftover opposite
> bit picking `kne2`/`knw2`. Overlays cast `w[em]` (caps included, faces never);
> a rim with both side bits where exactly one is bleed-borrowed is a turn and
> casts `rimE2`/`rimW2` toward the borrowed side.
>
> Round 4 also named five tiles no role could emit before: `rimE2 [0,4,12]`,
> `rimW2 [0,7,12]`, `capE2 [0,4,15]`, `capW2 [0,7,15]`, `faceBoth [0,8,16]`
> (brick = row − 10). An earlier round-3 derivation overfit the marked cells and
> changed 421 unmarked ones — the keeps tier exists because of it.

## Verification

The rule is verified before it is ported, never after. See `tests/js/`:

```
node tests/js/test_wall_rule.js     # the gate
node tests/js/mutation_check.js     # proves the gate bites
```

Run these **on the host**, where node lives. `make -C tests js` exists but the
`myfarm-builder` container has no node, so in Podman it skips with a notice rather
than running. Add `nodejs` to the Containerfile if you want one command for both.

Golden data is `tests/js/wall_cases.json`, generated by `gen_cases.js` from the
user's round-3 review. A review produces **two** kinds of evidence:

- **`fix`** — 72 cells the user marked, with the tile that should be there (56 we
  satisfy, 16 known-`xfail`). Solving an xfail is reported as news; rerun
  `gen_cases.js` to promote it into the gate.
- **`keep`** — 2796 cells in those same scenarios the user did **not** mark. An
  implicit *"this one is fine"*. They outnumber the fixes 39:1.

**The keeps are the whole ballgame.** The first version of this gate scored only
the fixes and generated its "approved" snapshots by running the *current* rule and
recording its output — comparing the rule to itself, so it always passed. Under
that gate the landed rule read **56/61 with 22/22 approvals preserved** while
actually changing **421 cells the user never asked to change**, 184 of them inside
renders the user had explicitly approved. The user caught it by eye; the suite
was structurally incapable of catching it.

So `keep` comes from `baseline_round3.js` — the rule exactly as it rendered when
the user was reviewing, recovered verbatim from the session transcript and
verified to reproduce **62/62** of the roles they recorded seeing. Never from the
current rule. Golden data that is generated from the thing under test cannot
contradict it.

`mutation_check.js` exists for the same reason: a green suite proves nothing until
you show it can go red.

**Score by drawn cell, never by role name.** Several roles alias the same tile
(`ksw`===`ccnw`===`[0,6,13]`, `kse`===`ccne`===`[0,5,13]`), so a name comparison
invents failures on tiles that render perfectly.

## Not yet in the game

`source/scenes/world_scene.cpp` still runs the **old** logic. The rule here is not
ported, by choice: it is verified first. Porting means extracting the geometry →
role decision into `source/core/wall_autotile.{h,cpp}` (pure, no `atlas_*_idx`, no
`<3ds.h>`) so `tests/` can reach it, leaving only role → atlas index in `drawMine`.
`wall_cases.json` is deliberately language-neutral so it can generate a
`tests/wall_cases.inc` and hold both implementations to the same cases.

Known-unsolved: 10 corrections want tiles no role emits (`(4,12) (7,12) (4,15)
(7,15) (8,16)`); 6 are the `kne2`/`knw2` variant-corner family. Floors were never
carefully reviewed — the `15 - openMask` under-wall default is unverified.
